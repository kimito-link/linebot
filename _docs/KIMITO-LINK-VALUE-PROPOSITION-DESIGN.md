## 経緯

「言葉で人格を自由に作れる」訴求でLPを作っていたが、実地調査でこの機能はコード未実装（りんくの人格は
`groq-knowledge-content.ts`にハードコードされた固定文言）と判明。一方dHash自己認識（送られた動画/画像が
キャラ自身の既知アセットと一致するか判定する機能）は実装済み・実機動作確認済み。

プロダクトオーナーの発言（原文）:「ビジネスでも、亡くなった個人でも。亡くなった個人の場合は、自分を認識して
もらえる。ビジネスも自分を認識してもらう。推し活やカスタマーサポート、汎用性は無限大と。それを設計したのでは。
それを有料サービスとして売ると」

マルチLLM会議ハーネス（groq/gpt-oss-120b・qwen3.6-27b・llama-3.3-70b、3体応答）＋Explore agentによる実地調査
（既存課金実装、dHashの技術的制約、LPコピーとの整合性、プライバシー記載、ビジネス向け既存実装の有無）を経て、
Fableに設計を委任した。

## 設計判断の要旨（結論）

1. **主軸は「自分を認識してもらえる」体験に一本化する。**「言葉で人格を作る」訴求は今日LPから削除する。
   機能としても当面実装しない。
2. **偲び・グリーフケア用途は提供しない。**LP・訴求・ロードマップのすべてから外す。現技術（登録済みアセット
   との近似一致）では「故人を認識する」体験は構造的に作れず、作れないものを遺族に売ることは倫理的に
   許容できない。
3. **料金は用途別2ライン。**全用途同一料金は捨てる。料金設計はdHashのGPU負荷ではなく**Gemini API課金と
   メディア判定回数**を軸に組む（C-1参照、コスト源の誤解を訂正）。

## A. 理想の体験フロー

### A-1. 推し活（主戦場、◎）

1. ファンがKimito Linkを友だち追加し、りんく/こん太/たぬ姉と日常トークする（無料、現行どおり）。
2. ファンが公式配布動画・画像をトークに送る。
3. Botが**AIを使わず決定的に**（Tier 0: SHA-256完全一致 → Tier 0.5: dHash近似一致）「それ、わたしの動画だ！」
   と一人称で反応する。
4. **価値実感の瞬間**: 「このキャラは自分のことを分かっている」という驚き。他のチャットボットには存在しない
   体験であり、実装済み・実機確認済み（`apps/worker/src/services/phash.ts`、`self-recognition.ts`）。
5. 有料化の接点: 無料はメディア判定を月数回まで、有料でメディア反応・限定アセット（有料会員限定動画=判定
   対象アセットの追加配布）を解放。「限定動画を受け取る→送り返すと特別な反応が返る」の往復がリテンション
   ループになる。

### A-2. ビジネス・カスタマーサポート（第2線、◯、ただし受注生産）

1. 企業が自社ブランドキャラ/ロゴ/製品カットを事前登録アセットとして預ける。
2. 顧客がLINEサポート窓口に製品写真・キャンペーン動画を送ると、Botが「その製品の話ですね」と文脈を掴んで
   サポートを開始する。
3. 価値実感: 「画像を送っただけで話が通じる」窓口体験。既存のknowledge-packs型サポートBot（ai-shain-link等
   の実績）に認識機能をアドオンする形。
4. **ただし流用できる既存実装は無い**（地雷#5）。LPで先売りせず、1社目は手動オンボードの受注生産で検証
   してから商品化する。

### A-3. 偲び・グリーフケア — **提供しない**

判断: **「今の技術でできる範囲に絞る」のでも「将来拡張前提で保留」でもなく、用途として扱わない。**

理由:
- 現実装で成立する体験は「生前に登録した特定動画を送り返すと一致する」だけであり、これは「故人が自分を
  認識してくれる」体験ではなく**「生前に用意した動画の再生装置」**に過ぎない。遺族が持つ未登録の写真・
  動画には一切反応できない。
- 遺族は「認識してくれた/くれなかった」を故人の意思の代理として受け取る心理状態にある。未登録写真への
  無反応は「忘れられた」体験として、偽陽性は「別人を故人と呼ぶ」体験として、いずれも通常ユースケースとは
  桁違いの心理的被害を生む。
- 文字どおり実現するには顔認識モデルへの全面置き換え（技術基盤の作り直し）+ 生前同意・遺族代表権限確認・
  専門家監修・要配慮個人情報の管理体制が必要で、個人開発1名の容量を構造的に超える。
- プロダクトオーナーの原発言のうち「亡くなった個人の場合は、自分を認識してもらえる」は**現プロダクトでは
  偽**であり、この一文を訴求に使うことを禁止事項とする。

## B. 統合アーキ

```
[コア資産] 公式アセット認識エンジン（1つだけ。共通実装）
  Tier 0   : SHA-256完全一致（決定的・コストゼロ）
  Tier 0.5 : dHash近似一致（決定的・純JS・コストほぼゼロ）※閾値較正中
  Tier 1   : Gemini客観描写 + 特徴テーブル照合（API課金あり、2026-07-21に403問題は解消済み）
      │
      ├── ライン1: Kimito Link（B2C・推し活）……現LPを改修して主力に
      └── ライン2: 認識アドオン付きサポートBot（B2B）……別LP・受注生産・後日
      ×── 偲びライン……作らない
```

会議の収束点どおり**1つのLPで全用途訴求はしない**。ただし分離の実体は「LPを2枚作る」ではなく、**今はライン1の
LP 1枚に絞り、ライン2はLPすら作らない**（受注実績ゼロの商品ページは誇大広告地雷の再生産になるため）。

## C. 具体機構

### C-1. コスト構造の正しい把握（前提の訂正）

会議の批判役（gpt-oss-120b）は「dHash処理のCPU/GPU負荷が増大し月額料金を超過する」と指摘したが、
**コスト源の特定が誤っている**。dHashはLINEプレビューJPEG1枚を9x8グレースケール化する純JS処理で、
1判定あたりのコストは実質ゼロ。実際のコストドライバーは:

1. **Gemini動画describe API**（Tier 1）— 従量課金が発生する唯一の箇所（2026-07-21時点で403問題は解消済み、
   `gemini-3.1-flash-lite`で稼働中）。
2. **アセット保存ストレージ** — 登録アセット数に比例。
3. （B2Bのみ）オンボード人件費 = オーナー本人の時間。

よって料金体系は「GPU負荷対策」ではなく**「Gemini呼び出し回数とアセット数に上限を切る」**ことで防御する。
同一料金の破綻リスクという結論自体は正しいので、以下のとおり分ける。

### C-2. 料金体系

| | 無料 | 推し活プラン | ビジネス（受注生産） |
|---|---|---|---|
| 価格 | ¥0 | **月額¥980** | **月額¥9,800〜 + 初期設定費**（個別見積） |
| トーク | 回数制限あり | 無制限 | 無制限 |
| メディア認識 | 月3回まで | 月30回まで | 月500回まで、超過は翌月繰越なしの追加パック |
| 登録アセット | 公式配布分のみ | 公式+限定配布分 | 50点まで預かり |
| Tier 1（Gemini） | 使わない | 上限内のみ | 上限内のみ |

- ¥980は既存プロダクト（soushin-suggest ¥980）と同帯で、決済は**外部LP→Stripe誘導方式を踏襲**
  （Bot内決済は作らない。実装コスト最小、既存Stripe Webhookレシーバーを流用）。
- 上限はすべて**fail-closed**: 超過時は「今月の認識回数を使い切ったのだ」と正直に返す。無言劣化させない。
- ビジネス価格はhenshin-hisho（¥2,980/月）より上に置く。理由はオンボードが手動であること、および安売り
  すると1社目の検証が赤字労働になること。

### C-3. dHash技術の使いどころの粒度

- **推し活**: 配布アセットは配布時点でdHashを事前計算しDB登録。受信時はTier 0→0.5で照合し、一致すれば
  **Gemini を呼ばずに**一人称反応を返す。つまり有料会員の中核体験がAPI費ゼロで回る。Tier 1は
  「未一致だが何か送ってくれた」時の贅沢品。
- **ビジネス**: 同一エンジンにテナント別アセット名前空間を足すだけ。ただし閾値はテナントごとに較正が要る
  （ロゴ・製品カットはキャラ動画サムネイルと分布が違う）ため、受注生産期間中に実測する。

### C-4. LP構成（ライン1）

1. ヒーロー: 「送った動画に、"それ、わたしだ"って気づいてくれる。」— 実装済み体験の断定表現。
2. 実演動画: 公式動画を送る→一人称で反応が返る実録（既存のdHashデモ動画をそのまま活用可能）。
3. 仕組みの正直な説明: 「公式の動画・画像を覚えていて、見分けます」（技術的に真）。
4. 料金表（C-2の2列のみ。ビジネスは「お問い合わせ」1行）。
5. 安心の話: 保存するもの・しないもの・期間・削除方法（D-3参照）。

## D. 偽陽性・誇大広告リスクを潰すロジック

### D-1. LPコピー修正方針（実在裏取り済み）

**結論: 人格カスタマイズ機能は実装しない。LPコピーから削除する。**両論の残し方（「将来実装予定」表記等）
もしない。

- 実装すれば誇大広告は解消するが、主軸（認識）と訴求が割れ、個人開発の実装容量を「まだ誰も課金していない
  機能」に食われる。認識体験は実装済みで今日から売れるのに対し、カスタマイズは競合（一般のキャラAI）との
  差別化にならない。
- 削除はコピー編集のみで即日完了し、景表法・特商法リスクが今日ゼロになる。

### D-2. 偽陽性（誤認識）対策

- 実装済みのfail-open原則を維持: 判定不能・複数キャラ同点はnullで通常応答（`self-recognition.ts`の設計
  どおり）。**「わからない時は黙る」がプロダクトの人格**。
- dHash閾値は較正完了（intra/inter距離の実測）まで、LP上で「必ず気づく」と書かない。「気づいてくれる」
  （頻度を約束しない表現）に留め、較正完了後に実測一致率を根拠として表現を強める。
- 有料プランの説明に「公式配布の動画・画像が対象です。お手持ちの一般の写真には反応しません」を明記。
  これが偲び用途の期待を入口で遮断する防波堤も兼ねる。

### D-3. プライバシー記載

「安心の話」セクションに追記する3点:
1. 送信された画像・動画の保存有無と保存期間（判定用ハッシュのみ保持し原本を保持しないなら、それを明記）。
2. 削除依頼の窓口と応答期限。
3. 顔認識・生体情報の特徴量化を**行っていない**ことの明示（dHashは画像全体の明暗パターンであり生体テンプレート
   ではない。これは倫理的な強みなので隠さず書く）。

| 案 | 出所 | 捨てる理由 |

## G. 地雷と回避策

1. **誇大広告**: 回避 = 本日のコピー修正（E）。以後のルール:「実機で動作確認した体験しかLPに書かない」。
   reality-checkerエージェントによる検証を経ていない機能表現はコミットしない。
2. **偲び期待の混入**: LPから用途を外しても、ユーザーが自発的に故人の写真を送るケースは起こり得る。
   回避 = fail-open設計（無反応→通常トーク）を維持し、認識対象を「公式配布アセット」とLP・ヘルプの双方に
   明記。故人写真に偽陽性で反応する事故だけは絶対に避けたいので、dHash閾値は保守側（偽陽性ゼロ寄り）に
   較正する。
3. **Gemini 403/429**: 2026-07-21時点で解消済み（`gemini-3.1-flash-lite`、新キー）。ただし予算アラート
   （月¥3,000）を設定済みなので再発時は早期検知できる。
4. **dHash閾値未較正**: 較正前に一致率を約束する表現をしない（D-2）。実測完了が「気づいてくれる」→
   強い断定への昇格条件。
5. **ビジネス用途の実績ゼロ**: 回避 = LPに商品ページを作らず「お問い合わせ」1行に留める。1社目を手動
   オンボードで納品し、テナント分離とdHash閾値のテナント別較正という2つの未知を実測してから商品化する。
6. **課金導線未実装**: 回避 = 外部LP+Stripe誘導の既存パターン踏襲。Bot側は「有料会員フラグ+回数上限」
   だけ持てばよく、既存Stripe Webhookレシーバーを流用できる。
7. **プライバシー記載不足**: 回避 = D-3の3点追記。特に「生体特徴量化をしていない」の明示は、認識訴求が
   強まるほど誤解（顔認識では？）が増えるため、訴求強化と同時に必須。

人格カスタマイズと実現不能な偲び用途を今日切り落とす。最初の一手はコード変更ゼロの`apps/lp/index.html`
コピー修正である。

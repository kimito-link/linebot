| サイズ上限 | `maxInputBytes = 15MB` | `bot.config.json` `llm.video`（実質の制約は締め切り側、B-1参照） |

**サイズ試算の基本式**:
```
ファイルサイズ[bytes] ≒ (映像bitrate + 音声bitrate)[kbps] × 1000 × 尺[秒] ÷ 8
例: 800kbps × 8秒 = 6400kbit = 800KB
```
`-maxrate` + `-bufsize`（VBVバッファ）を指定すればこの値が理論上の上限になる。

## A. LP用ヒーロー動画のffmpeg実装設計

### A-1. コンセプト: 「カメラを動かす」のをやめ、「キャラを生かす」

前回却下のzoompanは*カメラだけ*が動き、キャラは死んだ静止画のままだった（ユーザーからの「いまいち」評価の直接原因）。今回は**キャラクター空間の動き**（まばたき・口パク・表情変化・呼吸の上下動）を timestamp 精度で振り付ける。カメラはほぼ固定。

### A-2. 動きの文法（3種類を使い分ける）

| 動き | 遷移方法 | 理由 |
|---|---|---|
| まばたき | ハードカット（0.12〜0.16秒だけblink差分を重ねる） | 実物のまばたきは約100〜150ms。クロスフェードすると不自然 |
| 口パク | 周期的ハードカット（`mod(t,0.32)<0.16` で開閉交互） | ゆっくり系キャラの口パクはパタパタ切替が様式として正しい |
| 表情変化（normal→smile等） | アルファ・クロスフェード（0.3〜0.5秒） | 感情の変化だけは滑らかに繋ぐと「表情が動いた」と読める |
| 呼吸 | overlayのy座標を正弦波で±6px（周期4秒） | 常時微動があると静止区間も「生きて」見える |

### A-3. キャラ別ストーリーボード（8秒ループ、人格を動きに反映）

**りんく（対応役・明るい）**:

| 時刻 | 状態 | 遷移 |
|---|---|---|
| 0.0–2.0 | normal口閉じ（アイドル） | — |
| 2.0–2.14 | blink | ハードカット |
| 2.14–3.5 | normal口閉じ | — |
| 3.5–5.0 | 口パク（normal開⇄閉 0.16秒周期） | 周期enable |
| 5.0–7.4 | smile口閉じ | 0.35秒αフェードイン |
| 6.5–6.64 | blink（smileの上に重ねる） | ハードカット |
| 7.4–7.9 | smile→normalへ | 0.5秒αフェードアウト |
| 7.9–8.0 | normal口閉じ = 先頭と同じ絵 | ループ点整合 |

**こん太（判定役・クール）**: 口パク短め（4.0–4.8のみ）、smileの代わりにhalf-eyes（クール目）を5.5–7.0に挿入、まばたきは1回だけゆっくり(0.2秒)。呼吸振幅±4pxと控えめ。※talkペアは`kitsune-yukkuri-normal.png`(開)⇄`kitsune-yukkuri-mouth-closed.png`(閉)を使う（命名罠、地雷10参照）。

**たぬ姉（安全役・柔らかい）**: 0秒からずっとsmileベース、口パクはsmile開⇄閉でおっとり0.24秒周期、half-eyes（微笑の細目）を一瞬（5.0–5.6）、呼吸は周期5秒・±7pxと大きくゆったり。

呼吸の正弦周期はループ尺の約数にすること（周期が尺を割り切らないとループ点で高さが跳ぶ）。

### A-4. 実コマンド構造（りんく版）

Windowsの引用符地獄を避けるため `-filter_complex_script` を使う（地雷3参照）。

`fc-link.txt`:
```
color=c=0xFAF7F2:s=1500x1500:r=24:d=8[bg];
[0]format=rgba[base];
[1]format=rgba[talk];
[2]format=rgba,fade=in:st=5.0:d=0.35:alpha=1,fade=out:st=7.4:d=0.5:alpha=1[smile];
[3]format=rgba[blink];
[base][talk]overlay=0:0:enable='between(t,3.5,5.0)*lt(mod(t,0.32),0.16)'[s1];
[s1][smile]overlay=0:0[s2];
[s2][blink]overlay=0:0:enable='between(t,2.0,2.14)+between(t,6.5,6.64)'[s3];
[bg][s3]overlay=x=0:y='6*sin(2*PI*t/4)'[comp];
[comp]scale=480:480:flags=lanczos,fps=24,format=yuv420p[v]
```

```bash
ffmpeg -y \
  -loop 1 -t 8 -i link-yukkuri-normal-mouth-closed.png \
  -loop 1 -t 8 -i link-yukkuri-normal-mouth-open.png \
  -loop 1 -t 8 -i link-yukkuri-smile-mouth-closed.png \
  -loop 1 -t 8 -i link-yukkuri-blink-mouth-closed.png \
  -filter_complex_script fc-link.txt -map "[v]" \
  -c:v libx264 -profile:v main -crf 22 -maxrate 800k -bufsize 1600k \
  -movflags +faststart -an link-hero.mp4
```

**注意（司令塔追記）**: 上記コマンドの入力ファイル名は略記。実在パスは
`kimito-link/src/images/yukkuri-charactore-english/link/link-yukkuri-normal-mouth-closed.png`等、
フルパスに置き換えて実行すること。

構造のポイント:
- 合成は1500×1500のまま行い、最後に1回だけ480へ縮小（差分PNGは同一キャンバスで位置合わせ済みなので`overlay=0:0`で完全整合）。
- blinkを最上層に置く（smile表示中のまばたきでも違和感なし。smile用blink差分が無い問題の回避）。
- 呼吸は`overlay`のy式で実現（`t`参照可能、`scale=eval=frame`より古いffmpegでも壊れない）。
- 透過→黒背景問題は`color=c=0xFAF7F2`を最下層に敷く既確立の回避策をそのまま使用。

### A-5. 数値根拠

| パラメータ | 値 | 根拠 |
|---|---|---|
| 尺 | 8秒 | 会議収束点5〜10秒の中庸 |
| 解像度 | 480×480 | LP表示枠は約300px。480あればRetinaでも1.6x供給 |
| fps | 24 | 呼吸の正弦とフェードが滑らかに見える下限 |
| レート制御 | `-crf 22 -maxrate 800k -bufsize 1600k` | 理論上限=800kbps×8s÷8=800KB。実測見込み250〜600KB |
| 音声 | なし(`-an`) | `<video autoplay muted loop playsinline>`前提 |

→ 1〜3MB目標に対し理論上限0.8MB・実測見込み0.5MB前後で余裕をもって達成。

## B. LINE Bot送信テスト用動画の設計

### B-1. 本当のボトルネックは「サイズ上限15MB」ではなく締め切り連鎖

```
45秒(REPLY_DEADLINE) − 前段消費 ≥ 37秒(22s timeout + 15s margin)
⇒ describeVideo到達までの前段消費が 8秒以内 でなければ黙ってスキップされる
```

前段消費の内訳: webhook処理(~0.2s) + LINEトランスコード待ちポーリング(0〜最悪9秒) + コンテンツダウンロード(サイズ比例) + 雑処理。**ポーリング最悪値9秒だけで予算8秒を超える**。安全性は「Geminiが22秒で終わるか」より先に、LINE側トランスコードが1〜2ポーリング(≦3秒)で完了するほど動画が軽いかで決まる。

### B-2. Gemini側の負荷試算

Gemini 2.5 Flash Liteは動画を約1fpsでサンプリング。6秒動画≒6フレーム≒約1,500入力トークン。Flash Liteの推論は通常3〜8秒で、22秒タイムアウトに対し2.5倍以上のマージン。

### B-3. 推奨エンコード（LP版から`-b:v`等を変えて別出力）

```bash
ffmpeg -y \
  (入力・filter_complex_scriptはA-4と同じ構造。尺を6秒に、fade類の時刻を6秒尺に詰める) \
  -f lavfi -i anullsrc=r=44100:cl=mono \
  -map "[v]" -map 5:a -shortest \
  -c:v libx264 -profile:v baseline -level 3.0 \
  -b:v 350k -maxrate 450k -bufsize 900k -r 12 \
  -c:a aac -b:a 32k \
  -movflags +faststart link-bot-test.mp4
```

| パラメータ | 値 | 根拠 |
|---|---|---|
| 尺 | 6秒（絶対上限8秒） | Geminiサンプル6フレームで軽い |
| fps | 12 | Geminiは1fpsサンプルなので24fpsは無駄。口パク窓0.16秒≒2フレームでギリ発火 |
| ビットレート | `-b:v 350k -maxrate 450k` | 理論上限=(450+32)kbps×6s÷8≒362KB |
| プロファイル | baseline / level 3.0 / yuv420p | LINEトランスコーダに最も無難な入力 |
| 音声 | 無音AACトラック(32kbps) | 完全無音声mp4は稀に扱いが変わるための保険 |

**安全上限の明文化**: LINE Botに送るテスト動画は「尺≦8秒・ファイル≦1MB」を上限規約とする。

**やってはいけない調整**: `llm.video.timeoutMs`を延ばすこと。音声で実証済み（22→28秒でガード条件が実質43秒必要になり逆に全スキップ、実機で無反応を確認済み）。timeoutMsは22秒のまま動かさない。

## C. Bot反応設計 — 「自分自身の動画」を受け取ったとき

### C-1. 設計方針

現行パイプライン: Gemini が動画を客観描写 → その描写テキストを人格LLMがペルソナ返信に変換（画像で「人格が消えて生描写を返す」事故を877fb09で修正済み。動画も同じ轍を踏ませない）。

人格プロンプトに「自己外見カード」を持たせ、描写が自分に一致したら一人称で反応させる：

```
【自分と仲間の見た目】
- りんく: 金髪のゆっくり系キャラ。明るい笑顔が特徴
- こん太: 狐のゆっくり系キャラ
- たぬ姉: 狸のゆっくり系キャラ
送られた画像・動画の描写が上記に似ている場合、それは自分（または仲間）を
描いたファンアート・紹介動画である可能性が高い。その場合:
- 一人称で、少し照れ・喜び・ツッコミなど人格に沿った感情反応をする
- 「これは私の動画です」のような説明口調は禁止
- 確信が持てない描写なら「わたし…だよね?」と軽く確かめる形にする
- 動画の中の動き（まばたき・笑顔など）に1つだけ具体的に触れる
```

**重要**: Geminiは1fpsサンプルなのでまばたき(0.14秒)は描写に乗らない可能性が高い。反応セリフは「笑顔」「口の動き」など1秒以上続く要素に触れさせる。

### C-2. キャラ別反応例

**りんく（対応役・明るい）— 照れ+喜び**
> 「わっ、これ…わたしだ〜!✨ にこにこしてるとこ、ちょっと照れるけど嬉しいかも。ね、どこで見つけてくれたの?」

**こん太（判定役・クール）— 素直に喜ばない、品質を採点する**
> 「…俺か。ふん、口の動きの再現度は悪くないな。採点するなら85点。減点は照れ隠しだ、深く聞くな」

**たぬ姉（安全役・柔らかい）— 包み込む、少ししみじみ**
> 「あらあら、わたしの動画やねぇ…ふふ、こうして自分を外から見るのは不思議な気分。大事に持っててくれてありがとうねぇ」

### C-3. 避けるべきパターン

| アンチパターン | なぜダメか | 代わりに |
|---|---|---|
| 「これは私の動画です。私はりんくというキャラクターです」 | 機械的な自己説明。人格が死ぬ | 感情反応が先、説明はしない |
| 「え!?わたしが2人!?どういうこと!?」 | 混乱芸は台本臭い | 「外から自分を見るのは不思議」程度の軽い驚きまで |
| 「これはLPのヒーロー動画で、ffmpegで生成されています」 | 制作内幕のメタ漏れ | 世界観内で閉じる |
| 「かわいいキャラクターの動画ですね!」 | **最悪の失敗**。自分を他人事として描写＝画像事故(877fb09修正前)の再演 | 自己外見カードで一人称化 |
| 「これは間違いなく私です」 | Gemini誤認リスクに対し確信を断言すると外したとき痛い | ヘッジ表現（〜だよね?/〜な気がする） |

## E. 捨てた案と理由

1. **全遷移xfade/クロスフェード**: まばたき・口パクをフェードすると目や口が半透明に溶ける。前回の二の舞。→ハードカット併用に変更。
2. **パーツ分解版でのリグ的アニメ**: 表現力は最高だがfilter_complexが組合せ爆発。フルフェイス差分スワップで体感品質の8割が2割のコストで出る。将来の温存カード。
3. **透過WebM(VP9 alpha)でLP背景に直接載せる**: SafariがVP9 alpha非対応。背景色焼き込みで同じ見た目が出るため却下。
4. **GIF/APNG**: 同品質でmp4の5〜10倍のサイズ。
5. **LP動画15秒案**: 帯域2倍、振り付けネタ切れでループ疲れ。
6. **LP動画をそのままBotテストに流用**: fps24はGeminiの1fpsサンプルに対して無駄。別出力(6秒/12fps/~300KB)を正とする。
7. **`llm.video.timeoutMs`延長で安全マージン確保**: 音声の実障害で反証済み。

## F. 地雷と回避策

1. **透過PNG→黒背景**: `color=c=0xFAF7F2`を最下層に敷く既確立回避策を全コマンドで踏襲。
2. **奇数解像度でlibx264が失敗**: `yuv420p`は偶数辺必須。
3. **Windows引用符地獄**: `filter_complex`内の`'between(t,...)'`はPowerShellで確実に壊れる。→`-filter_complex_script`でASCIIパスのファイルに外出し、Git Bashで実行。素材パスに日本語(デスクトップ)が含まれるためscratchpadへコピーしてから作業。
4. **fadeのalpha指定漏れ**: `fade=in:...`に`alpha=1`を付け忘れると黒フェードになる。前段の`format=rgba`も必須。
5. **`-loop 1`入力の`-t`忘れ**: 無限エンコードで固まる。全PNG入力に`-t 尺`を明示。
6. **口パク窓とfpsの関係**: enable窓(0.16秒)が`1/fps`を下回ると口パクが1フレームも発火しない。
7. **ループ点の跳び**: 呼吸正弦の周期が尺を割り切らない/フェードアウトが最終フレームまでに完了しないとループ瞬間に絵がポップする。
8. **Geminiは1fpsサンプル**: まばたき0.14秒は描写に乗らない（C-1で対処済み）。
9. **タイムアウト連鎖**: 真の予算は「describeVideo到達まで8秒」。防御は入力サイズ(≦1MB/≦8秒)のみ。timeoutMs=22秒は動かさない。
10. **こん太の命名罠**: `kitsune-yukkuri-normal.png`/`kitsune-yukkuri-mouth-closed.png`が口開閉ペア（他キャラの`normal-mouth-open/closed`に相当）。3キャラ分の入力マッピング表をスクリプト冒頭に明記すること。
11. **`-movflags +faststart`忘れ**: moovアトムが末尾のままだとVercel配信でLP動画の再生開始が遅れる。全出力に付与。
12. **OneDrive同期ロック**: デスクトップ配下で直接ffmpeg出力するとファイルロック/同期競合の恐れ。→scratchpadで生成し、完成品だけ`apps/lp/assets/video/`へコピー。
13. **LINEアプリ側の再圧縮**: スマホから送る際、カメラロール経由だとLINEが再圧縮して当方のエンコードが変わりうる。テスト時は送信経路も記録して再現性を確保する。

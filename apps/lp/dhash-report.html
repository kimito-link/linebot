Gemini が落ちても、動画のキャラ認識だけは止めなかった話

ロミさんの「動画は画像に変換してから見せる」という話をヒントに、 AI を一切使わずに動画を認識する方法を試してみました。 知覚ハッシュ（dHash）というやつです。LINE の再エンコードを経ても、 ローカルの元データと実機データの両方で確かめられたので、記録しておきます。

対象 LINE Bot（Cloudflare Workers） 状況 Gemini API が請求先未設定で 403 手段 dHash（差分ハッシュ）+ LINE preview API

きっかけ

うちの LINE Bot は、ユーザーが送ってきた動画を Gemini に見せて「これは自分自身の動画だ」と 気づかせる機能を持っています。ところが Gemini 側の請求先アカウントが未設定で、 403 PERMISSION_DENIED のまま完全に沈黙してしまいました。

ロミさんの投稿で「動画は ffmpeg で画像数枚に切り出してから Qwen に見せている」という 運用を知り、動画をそのまま AI に渡す以外の道があることに気づきました。 Cloudflare Workers は ffmpeg が使えない環境なので同じ手は使えませんが、 発想だけ借りて別の実装を試すことにしました。

やったこと：動画を「見ず」に、写真として当てる

LINE の Messaging API には、動画メッセージ専用の GET /content/preview というエンドポイントがあります。動画本体ではなく、 その動画のサムネイル JPEG だけを返してくれるものです。

Bot が送っている動画はもともと 8 秒のキャラクタールーープ動画で、 全編ずっと同じキャラが写っています。だとすれば 1 枚のサムネイルだけで、 動画に写っているキャラを言い当てるのに十分 なはずだと考えました。

あとはこのサムネイルを「知覚ハッシュ（dHash）」という指紋に変換し、 Bot 自身が送った動画のサムネイルとどれだけ近いかを測るだけです。 AI 抜きの、ただの数値比較です。

知覚ハッシュとは

SHA-256 のような暗号学的ハッシュは、1 ビットでも違えば全く別の値になります。 LINE は送信時に動画を必ず再エンコードするので、元ファイルとの完全一致判定（Tier 0）は ほとんど当たりませんでした。

dHash はその逆で、見た目が近ければ値も近くなる よう作られたハッシュです。 仕組みはシンプルで、画像を 9×8 ピクセルまで縮小してグレースケール化し、 横に隣り合うピクセルの明暗を比べて 0/1 を並べるだけ。64 ビット（16 進数で 16 文字）の 指紋になります。

2 つの画像がどれだけ似ているかは、この 64 ビットのうち何ビット違うか （ハミング距離）で測れます。同じ画像なら距離 0、全く違う画像なら距離は大きくなります。

同一キャラの距離は最大でも 8、別キャラの距離は最小でも 18。 間に 10 ビットの余白があるので、「距離 10 以下なら同一キャラ」という閾値を 安全に引けます。合成テストではなく本物のキャラクター画像でこの結果が出たのは大きな収穫でした。

実測 2：本番で実際に動画を送ってみる

ローカルのオリジナルファイルで良い数字が出ても、LINE を経由した実際のデータでも 同じことが言えるかは別問題です。本番の Bot に観測用ログだけを仕込んでデプロイし、 たぬ姉の動画を実際に LINE から送信しました。

そのまま Tier として組み込んだ

元々このBotの自己認識は 2 段構えでした。まず動画のバイト列が Bot 自身の送信済み動画と 完全一致するか（SHA-256）を見て、当たらなければ Gemini に説明文を書かせて キーワードでキャラを判定する、という流れです。

今回の dHash は、この 2 つの間に差し込みました。

Tier手段AIGemini 障害の影響
0SHA-256 完全一致なし受けない
0.5dHash 近似一致（今回）なし受けない
1Gemini describe + キーワード判定あり受ける

複数キャラが同じ最小距離で並んだ場合は、誤って言い当てるより黙って Tier 1 に フォールバックするようにしています（fail-open）。1 枚のサムネイルしか見ていないので、 「まばたきしてましたね」のような動きの描写は生成させないようにも調整しました。

ロミさんの方式との違い

ロミさんの「動画→画像に変換してから見せる」というアプローチは同じ発想の起点でした。 違うのは変換の手段と、その先で AI を使うかどうかです。

  • ロミさん方式 — ffmpeg で複数フレームを抜き出し、Qwen（画像認識 AI）に見せる。 情報量が多く、動画中盤の変化も拾える。ffmpeg が動く環境が前提。
  • 今回の方式 — LINE 標準の preview API で 1 枚だけ取り、AI を介さず数値比較する。 Cloudflare Workers のように ffmpeg が使えない環境でも動く。ただし全編で同じ絵が続く動画向けの割り切り。
まとめると

「動画をそのまま AI に渡さない」という一段目の発想はロミさんに教わったもので、 今回はそこからさらに一歩、AI 自体を使わない決定的判定まで踏み込んでみた、という位置づけです。
